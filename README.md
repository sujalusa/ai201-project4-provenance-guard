# Provenance Guard

A content-attribution service. A creator submits text; the system estimates
whether it was written by a human or generated by AI, reports **how confident**
it is (never a bare binary), shows the reader a plain-language transparency
label, and records every decision in an append-only audit log. Creators can
contest a verdict through an appeals workflow.

Full design rationale lives in [planning.md](planning.md). This README is the
operator/grader's guide: how to run it, what each feature does, and the evidence
that it works.

---

## Architecture

```
POST /submit ─► rate limiter ─► validate ─┬─► Signal 1: stylometry ─┐
                                          └─► Signal 2: Groq LLM  ──┤
                                                                    ▼
                                    fusion (agreement-weighted) ─► confidence
                                                                    ▼
                                    transparency label ─► audit log ─► response

POST /appeal ─► look up content_id ─► status = under_review ─► audit log ─► response
```

On submission, raw text passes the rate limiter and validator, then fans out to
both detection signals; fusion combines them into a combined score and an
agreement-weighted confidence, which the label builder turns into one of three
reader-facing variants, and the whole decision is appended to the audit log
before the response returns. On appeal, the creator's `content_id` and reasoning
flip the content's status to `under_review` and append a linked appeal entry to
the same log for a human to resolve.

**Modules:** [stylometry.py](stylometry.py) (Signal 1) ·
[llm_signal.py](llm_signal.py) (Signal 2) · [fusion.py](fusion.py) (confidence
scoring) · [labels.py](labels.py) (transparency labels) · [store.py](store.py)
(mutable content state) · [audit.py](audit.py) (append-only log) ·
[app.py](app.py) (Flask API).

### API

| Method & path | Accepts | Returns |
|---|---|---|
| `POST /submit` | `{ "text", "creator_id" }` | `{ content_id, attribution, confidence, signals:{stylometry, llm}, combined_score, label_text, status }` |
| `POST /appeal` | `{ "content_id", "creator_reasoning" }` | `{ content_id, status:"under_review", appeal_logged:true }` |
| `GET /log` | optional `?content_id=` / `?status=` | `{ entries: [...] }` |
| `GET /health` | — | `{ status:"ok" }` |

`attribution ∈ {likely_ai, likely_human, uncertain}`. Errors: `400`
missing/short text or empty reason, `404` unknown `content_id`, `429` over rate
limit.

## Detection signals (multi-signal pipeline)

Two **genuinely independent** signals — one structural, one semantic. They fail
in different ways, so their disagreement is itself information.

### Signal 1 — Stylometric heuristics (structural, pure Python)
Measures statistical regularity of writing; AI trends uniform, humans bursty.
Four sub-metrics, each normalized to an AI-likeness in `[0,1]` and weighted:
- **Sentence-length CV** — rhythm/burstiness (low variance → AI).
- **Type-token ratio** — vocabulary diversity (flat/low → AI).
- **Punctuation variety** — stylistic texture (few kinds → AI).
- **Boilerplate-phrase density** — lexical AI tells ("It is important to note",
  "Furthermore", "paradigm shift"…). Weighted highest (0.40) because it is the
  only metric that catches polished AI prose whose sentence variance and
  vocabulary otherwise read as human. Its absence is only weak evidence of a
  human, so the weight is asymmetric by design.

**Blind spot:** content-blind. Plain/uniform human writing can score AI-like;
AI told to "write bursty" without boilerplate mimics human structure;
unreliable on very short text.

### Signal 2 — LLM classification via Groq (semantic)
`llama-3.3-70b-versatile`, `temperature=0`, JSON-mode. Holistic judgment of
whether text *reads* as machine-generated (clichés, hedging, generic transitions
vs. lived specificity). Returns `ai_probability ∈ [0,1]` + one-sentence
reasoning; on any API/parse error it returns `None` and fusion falls back to
stylometry with a penalty.

**Blind spot:** non-deterministic, can be overconfident, biased against
non-native-English/unusual-but-human styles, defeatable by paraphrase.

**Why this pairing:** stylometry is local, deterministic, cheap but shallow; the
LLM is deep but noisy and remote. Agreement → confidence; disagreement → the
verdict is pulled toward "uncertain" instead of a false binary.

## Confidence scoring with uncertainty

Confidence is **not** "probability of AI" — it is *how much to trust the
verdict*. Direction and strength are computed separately, so the label is **not
a binary flip at 0.5**.

```
C  (combined)     = 0.6*llm + 0.4*stylometry          # direction: >=0.5 AI, <0.5 human
A  (agreement)    = 1 - |llm - stylometry|
D  (decisiveness) = min(1, 2.5*|C - 0.5|)
confidence        = D * (0.6 + 0.4*A)                 # agreement scales 0.6x..1.0x
```

Bands: `confidence ≥ 0.45` → classified (AI/human by direction); otherwise
**uncertain**. The agreement term *scales* rather than *multiplies to zero*, so a
confident LLM with only partial stylometry support still earns moderate
confidence.

### How I tested that the scores are meaningful
Calibrated in Milestone 4 against the four reference inputs (clearly-AI,
clearly-human, and two borderline cases), printing both signals separately:

| Input | stylometry | llm | combined | **confidence** | verdict |
|---|---|---|---|---|---|
| clearly AI | 0.57 | 0.80 | 0.71 | **0.48** | likely_ai |
| clearly human | 0.11 | 0.20 | 0.17 | **0.81** | likely_human |
| formal human (borderline) | 0.27 | 0.80 | 0.59 | **0.18** | uncertain |
| lightly-edited AI (borderline) | 0.27 | 0.70 | 0.53 | **0.06** | uncertain |

The confident cases (0.48, 0.81) sit well clear of the borderline cases (0.18,
0.06); the 0.45 threshold falls in that gap with margin, so score jitter won't
flip a verdict. The first calibration attempt put **all four at "uncertain"** —
diagnosing it (stylometry under-read polished AI; `D*A` collapsed on
disagreement) led to the boilerplate metric and the non-collapsing formula
above. Reproduce with `python test_signals.py`. A 0.51 vs 0.95 result therefore
produces meaningfully different labels (different band and/or wording), as shown
next.

## Transparency label (three variants)

The label returned by `/submit` changes with the confidence score and signal
agreement. `{pct}` = `round(confidence*100)`. Exact text of each variant:

**HIGH-CONFIDENCE HUMAN** (verdict human, `confidence ≥ 0.45`)
> ✍️ Likely human-written — 81% confidence. Our analysis strongly indicates this
> text was written by a person. We found the natural variation and voice typical
> of human writing. This is an automated assessment, not a guarantee.

**HIGH-CONFIDENCE AI** (verdict ai, `confidence ≥ 0.45`)
> 🤖 Likely AI-generated — 48% confidence. Our analysis indicates this text was
> produced by an AI system. Both our writing-pattern check and our language-model
> reviewer point the same way. This is an automated assessment, not a certainty —
> if you're the creator and believe it's wrong, you can appeal.

**UNCERTAIN** (`confidence < 0.45`)
> ❓ Origin uncertain — 18% confidence. We couldn't reliably determine whether
> this text was written by a human or generated by AI, though it leans slightly
> toward AI. Our two checks disagreed with each other. Please treat this as
> inconclusive — it isn't a judgment about the creator, who can request a review.

Wording adapts to be honest, not just to swap a label:
- **Strength** — "strongly indicates" at `confidence ≥ 0.75`, else "indicates",
  so a 48% AI result doesn't overclaim like an 81% one.
- **Support** (AI variant) — describes both signals concurring only when
  `agreement ≥ 0.70`; otherwise credits the LLM "with partial support".
- **Lean / agreement** (uncertain variant) — states which way it leans (from
  `C`) and whether the checks "disagreed" (`A < 0.5`) or just "weren't decisive".

The AI and uncertain variants never read as an accusation and always point to
the appeal path — this is how false positives are kept safe.

## Appeals workflow

- **Who:** the creator, identified by the `content_id` from their `/submit`
  response (no auth in this project — possession of the id is standing).
- **Provide:** `{ "content_id", "creator_reasoning" }` (reasoning required).
- **On receipt:** look up the original decision (`404` if unknown) → set the
  content's `status` to **`under_review`** → append a linked `appeal` entry to
  the audit log (with the reasoning + original verdict/confidence) → return
  confirmation. No automated re-classification; a human resolves it.
- **Reviewer view:** `GET /log?status=under_review` surfaces the appealed items
  with the creator's reasoning alongside the original decision.

```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "PASTE-ID", "creator_reasoning": "I wrote this myself..."}'
# -> {"appeal_logged": true, "content_id": "...", "status": "under_review", ...}
```

## Rate limiting

Applied with Flask-Limiter (per-IP, in-memory store).

| Endpoint | Limit | Reasoning |
|---|---|---|
| `POST /submit` | **10/minute; 100/day** | A real creator submits a handful of pieces in a sitting — 10/min covers editing-and-resubmitting a draft comfortably, while stopping a script from flooding the endpoint (each call costs a Groq request). 100/day bounds sustained cost/abuse without limiting genuine daily use. |
| `POST /appeal` | **20/hour** | Appeals are rare, deliberate actions; 20/hour is generous for a legitimate creator contesting several verdicts, yet blocks automated appeal spam. |

**Evidence** — 12 rapid `/submit` requests against a 10/min limit:
```
200 200 200 200 200 200 200 200 200 200 429 429
```
First 10 succeed; the rest return `429 Too Many Requests`.

## Audit log

Every decision and every appeal is one structured JSON line in
`audit_log.jsonl` (append-only), surfaced via `GET /log`. Each **decision**
entry records: timestamp, content_id, creator_id, attribution, confidence,
combined_score, **both individual signal scores** (`stylometry_score`,
`llm_score`) + raw stylometry metrics, agreement, the label text, and
`appeal_filed`. Each **appeal** entry records the reasoning, the original
verdict/confidence, and `status: under_review`.

## Known limitations / edge cases
- **Minimalist/repetitive human writing** can look AI-like structurally; the LLM
  usually disagrees → uncertain + appeal (the designed safety case).
- **Adversarial AI** told to write bursty and avoid boilerplate defeats
  stylometry; the system then leans on the LLM alone → typically uncertain.
- **Non-native-English human writing** may be flagged by the LLM; stylometry's
  human-leaning read pulls the result to uncertain rather than a confident false
  positive.
- **Short text** (< 8 words) is rejected; just above that, thin evidence keeps
  confidence low.
- Rate-limit and content state are in-memory / single-file JSON — fine for this
  project, not for multi-process production.
