# Provenance Guard — Specification & Plan

Provenance Guard is a content-attribution service. A creator submits text; the
system estimates whether it was written by a human or generated by AI, reports
**how confident** it is, shows the reader a plain-language transparency label,
and records every decision in an append-only audit log. Creators can contest a
verdict through an appeals workflow.

This document is both the design spec and the prompting tool used to generate
implementation code in Milestones 3–5. Every number here is meant to be
implementation-ready.

---

## 1. Detection Signals

Two **genuinely independent** signals — one structural, one semantic — combined
with an agreement-weighted fusion. They fail in different ways, so disagreement
between them is itself information.

### Signal 1 — Stylometric heuristics (structural, pure Python)
Measures statistical regularity of the writing. AI text trends *uniform*; human
text trends *bursty/varied*.

Four sub-metrics, each mapped to an AI-likeness in `[0,1]` (higher = more
AI-like):

| Sub-metric | What it captures | Human range (approx) | Maps to AI-like when… |
|---|---|---|---|
| Sentence-length CV (coefficient of variation = stdev/mean of sentence word-counts) | rhythm / burstiness | ~0.4–0.8 | CV is **low** (uniform) |
| Type-token ratio (unique words ÷ total, on first 200 tokens) | vocabulary diversity | ~0.55–0.75 | TTR is **mid/low + flat** |
| Punctuation variety (distinct punctuation marks used: `, ; : — ? ! ( )`) | stylistic texture | 4+ distinct | **few** distinct marks |
| Boilerplate-phrase density (regex word-list of AI-favored transitions/hedges per sentence) | lexical AI tells | ~0 | **many** boilerplate phrases |

Each sub-metric is normalized to `[0,1]` AI-likeness, then **weighted** (the
boilerplate metric carries the most weight — it's the only one that catches
polished AI text whose sentence variance and vocabulary read as human):
```
sty_var    = clamp((0.50 - sentence_cv) / 0.50, 0, 1)         # low CV -> AI
sty_ttr    = clamp((0.65 - ttr)        / 0.40, 0, 1)          # low diversity -> AI
sty_punct  = clamp((4   - punct_kinds) / 4.0,  0, 1)          # few kinds -> AI
sty_boiler = clamp(boilerplate_hits / n_sentences / 0.75, 0, 1)  # many tells -> AI
stylometry_score = 0.25*sty_var + 0.20*sty_ttr + 0.15*sty_punct + 0.40*sty_boiler
```
**Output shape:** `{ "score": 0.0-1.0, "metrics": { "sentence_cv", "ttr",
"punct_kinds", "boilerplate_hits", "n_sentences", "n_tokens" } }` — raw metrics
are kept for the audit log.

**Why boilerplate is weighted highest:** calibration (M4) showed canonical AI
prose (varied sentences, rich vocabulary) defeats the structural metrics; its
"AI-ness" is lexical ("It is important to note", "Furthermore", "paradigm
shift"). Absence of boilerplate is only weak evidence of human authorship, so
its high weight is asymmetric-by-design: it lifts AI text without dragging plain
human text up (humans rarely use these phrases).

**Blind spot:** content-blind. A deliberately plain human poem scores AI-like;
AI told to "write bursty" without boilerplate can mimic human structure.
Unreliable on very short text (too few sentences to compute variance).

### Signal 2 — LLM classification via Groq (semantic)
Model: **`llama-3.3-70b-versatile`** (configurable via `GROQ_MODEL` env var),
`temperature=0` for stability. Holistic judgment of whether the text *reads* as
machine-generated — clichés, hedging, generic transitions vs. lived specificity.

Prompt asks for strict JSON back:
```json
{ "ai_probability": 0.0-1.0, "reasoning": "one sentence" }
```
`llm_score = ai_probability` (higher = more AI-like).

**Output shape:** `{ "score": 0.0-1.0, "reasoning": "..." }`. On API error or
unparseable output, the signal returns `{ "score": null, "reasoning": "error" }`
and fusion falls back to stylometry alone with a confidence penalty.

**Blind spot:** non-deterministic, can be overconfident, biased against
non-native-English or unusual-but-human styles, defeatable by paraphrase, and
costs a network call.

### Fusion (agreement-weighted)
```
C  (combined_score) = 0.6 * llm_score + 0.4 * stylometry_score     # in [0,1]
A  (agreement)      = 1 - abs(llm_score - stylometry_score)        # 1=agree, 0=disagree
D  (decisiveness)   = min(1, 2.5 * abs(C - 0.5))                   # 0 at boundary, 1 when clearly off it
confidence          = round(D * (0.6 + 0.4 * A), 2)               # in [0,1]
```
LLM is weighted higher (0.6) as the deeper signal; stylometry (0.4) anchors it
with a deterministic check. **Confidence is suppressed both when the signals
disagree (low A) and when the combined score sits near the boundary (low D).**
The agreement term scales the result between **0.6×** (full disagreement) and
**1.0×** (full agreement) rather than multiplying straight by `A` — calibration
(M4) showed pure `D*A` collapses confidence to near-zero whenever the noisier
stylometry signal disagrees with the LLM, even on text the LLM reads clearly.
The `2.5` decisiveness gain keeps confident-but-imperfect-agreement cases (where
combined never reaches the extremes) from being under-scored. If the LLM signal
errored, the agreement factor is fixed at 0.6 (single-signal penalty).

## 2. Uncertainty Representation

**What confidence means:** it is *not* "probability of AI" — it is "how much
should you trust this verdict." A confidence of **0.18** means the evidence
leans one direction but the signals disagree and/or the margin is thin → it
lands in the **uncertain** band with directional phrasing; **0.81** means the
signals strongly agree on a decisive verdict.

**Direction vs. strength are separate:** direction comes from `C` (≥0.5 → AI,
<0.5 → human); strength comes from `confidence`. This is why the label is **not
a binary flip at 0.5** — crossing 0.5 only changes the *direction*; whether a
result is shown as "high-confidence" or "uncertain" depends on the confidence
band.

**Three bands → three label variants:**

| Band | Condition | Variant shown |
|---|---|---|
| Likely AI | `confidence ≥ 0.45` **and** `C ≥ 0.5` | HIGH-CONFIDENCE AI |
| Likely human | `confidence ≥ 0.45` **and** `C < 0.5` | HIGH-CONFIDENCE HUMAN |
| Uncertain | `confidence < 0.45` | UNCERTAIN (text adapts: "leans AI/human") |

Worked examples (measured in M4 against the milestone test set):
- `clearly AI`: llm=0.80, sty=0.57 → C=0.71, A=0.77, D=0.52, **conf=0.48** → Likely AI.
- `clearly human`: llm=0.20, sty=0.11 → C=0.17, A=0.91, D=0.84, **conf=0.81** → Likely human.
- `formal human` (false-positive case): llm=0.80, sty=0.27 → C=0.59, A=0.47, D=0.22, **conf=0.18** → Uncertain, leans AI.
- `lightly-edited AI`: llm=0.70, sty=0.27 → C=0.53, A=0.57, D=0.07, **conf=0.06** → Uncertain, leans AI.

**Why the 0.45 threshold:** measured confidences separate cleanly — clearly-AI
0.48 and clearly-human 0.81 sit well above the two borderline cases (0.18, 0.06).
0.45 sits in that gap with margin on both sides, so small score jitter won't flip
a verdict. Note clearly-AI lands at a *moderate* 0.48 (not 0.9+): stylometry only
partly corroborates the LLM there, and the label wording (§3) scales with the
percentage so it reads as "likely," not "certain."

**Calibration approach:** stylometry thresholds were seeded from typical
human/AI ranges, then validated in M4 against the milestone's labeled samples
(clearly-AI, clearly-human, two borderline). Success criterion (met): clearly-AI
→ AI verdict, clearly-human → human verdict, both borderline → uncertain, with a
wide confidence gap between the confident and uncertain cases. When the textbook
AI sample initially scored too low, the fix was a new stylometry sub-metric
(boilerplate density) plus a non-collapsing confidence formula — not moving the
band threshold to paper over it.

## 3. Transparency Label Design

Three variants. `{pct}` = `round(confidence * 100)`. Text is plain-language and
written so the uncertain/AI variants never read as an accusation.

**HIGH-CONFIDENCE AI**
> 🤖 **Likely AI-generated** — {pct}% confidence.
> Our analysis strongly suggests this text was produced by an AI system. Both
> our writing-pattern check and our language-model reviewer agreed. This is an
> automated assessment, not a certainty — if you're the creator and believe
> this is wrong, you can appeal.

**HIGH-CONFIDENCE HUMAN**
> ✍️ **Likely human-written** — {pct}% confidence.
> Our analysis strongly suggests this text was written by a person. We found the
> natural variation and voice typical of human writing. This is an automated
> assessment, not a guarantee.

**UNCERTAIN**
> ❓ **Origin uncertain** — {pct}% confidence.
> We couldn't reliably determine whether this text was written by a human or
> generated by AI{lean_clause}. Our two checks {agreement_clause}. Please treat
> this as inconclusive — it is not a judgment about the creator, who can request
> a review.

Where, for the uncertain variant:
- `lean_clause` = `", though it leans slightly toward AI"` if `C ≥ 0.5` else
  `", though it leans slightly toward human authorship"`.
- `agreement_clause` = `"disagreed with each other"` if `A < 0.5` else
  `"weren't decisive enough to be sure"`.

This makes a 0.62 result read differently from a 0.51 result (different pct +
lean wording) and both read differently from a 0.95 high-confidence result.

## 4. Appeals Workflow

- **Who can appeal:** the creator of a submitted piece, identified by the
  `content_id` returned at submission. (No auth in this project; possession of
  the id is treated as standing.)
- **What they provide:** `{ "content_id": "...", "reason": "<their reasoning>" }`.
  `reason` is required and non-empty.
- **What the system does on receipt:**
  1. Look up the original decision by `content_id` (→ `404` if unknown).
  2. Set the content's **`status` → `"under_review"`**.
  3. Append an **appeal entry** to the audit log, linked to the original
     decision record (`type: "appeal"`, includes the reason and a timestamp).
  4. Return `{ content_id, status: "under_review", appeal_logged: true }`.
  - No automated re-classification — a human resolves it.
- **What a human reviewer sees in the appeal queue:** for each appealed item —
  the original text excerpt, the verdict + confidence, both signal scores and
  raw stylometry metrics, the LLM reasoning, the creator's appeal reason, and
  the timestamps of both the decision and the appeal. (Surfaced via
  `GET /log?status=under_review` or `GET /log`.)

## 5. Anticipated Edge Cases

Specific cases this system handles poorly, and how the design copes:

1. **Minimalist / repetitive poetry** (short even lines, simple vocabulary,
   intentional repetition). Stylometry sees low sentence-CV + low TTR and scores
   it AI-like, but the LLM often senses genuine voice → signals disagree →
   confidence collapses → **uncertain** label + appeal path. This is the
   designed false-positive safety case.
2. **Very short submissions (< ~3 sentences / < 40 words).** Not enough
   sentences to compute meaningful variance; stylometry is noise. Mitigation:
   enforce a minimum length at `/submit` (`400` if too short) and, just above
   the floor, the thin-evidence low decisiveness keeps confidence low.
3. **Adversarial AI text prompted to "write bursty, varied, human-like."**
   Defeats stylometry by design; the system then leans almost entirely on the
   LLM, and agreement-weighting means a lone LLM call yields only moderate
   confidence → often **uncertain**. Documented as a known limitation.
4. **Non-native-English human writing.** The LLM may flag fluent-but-atypical
   phrasing as AI (bias). Stylometry (high human variance) can pull the other
   way → disagreement → uncertain rather than a confident false positive.
5. **Hybrid text** (human draft, AI-polished or vice versa). Genuinely
   ambiguous; the system is *expected* to return uncertain here, which is the
   honest answer.

## Architecture

### Submission flow
```
client
  │  raw text (JSON)
  ▼
POST /submit ──(over rate budget?)──► 429 Too Many Requests
  │ within budget
  ▼
validate text ──(missing / too short)──► 400 Bad Request
  │ valid raw text
  ├──────────────► Signal 1: Stylometry ──► stylometry_score [0,1] + raw metrics
  │                                              │
  ├──────────────► Signal 2: Groq LLM ─────► llm_score [0,1] + reasoning
  │                                              │
  ▼                                              ▼
        Fusion: C=0.6·llm+0.4·sty, A, D  ◄── both signal scores
  │ combined_score C + confidence (= D·A)
  ▼
Label builder (band by confidence + direction) ──► label_text (1 of 3 variants)
  │ verdict + confidence + label_text
  ▼
Audit log (append decision record) ──► scores, metrics, verdict, confidence, label
  │
  ▼
response: { content_id, verdict, confidence, signals, combined_score, label_text, status }
```

### Appeal flow
```
client
  │  { content_id, reason }
  ▼
POST /appeal ──(unknown id)──► 404 ; (empty reason)──► 400
  │ found original decision
  ▼
update content status ──► "under_review"
  │ original record + reason
  ▼
Audit log (append appeal entry, linked to original decision)
  │
  ▼
response: { content_id, status: "under_review", appeal_logged: true }
```

**Narrative:** On submission, raw text passes the rate limiter and validator,
then fans out to both signals; fusion combines them into a combined score and an
agreement-weighted confidence, which the label builder turns into one of three
reader-facing variants, and the whole decision is appended to the audit log
before the response returns. On appeal, the creator's `content_id` and reasoning
flip the content's status to `under_review` and append a linked appeal entry to
the same audit log for a human to resolve.

### API surface (contract)
| Method & path | Accepts | Returns |
|---|---|---|
| `POST /submit` | `{ "text": "<content>" }` | `{ content_id, verdict, confidence, signals:{stylometry, llm}, combined_score, label_text, status }` |
| `POST /appeal` | `{ "content_id", "reason" }` | `{ content_id, status:"under_review", appeal_logged:true }` |
| `GET /log` | optional `?content_id=` / `?status=` | array of structured audit entries (decisions + appeals) |
| `GET /health` | — | `{ status:"ok" }` |

- `verdict ∈ {"ai","human","uncertain"}`; `confidence ∈ [0,1]`.
- Rate limiting on `POST /submit` (and `POST /appeal`). Errors: `429` over
  limit, `400` missing/short text or empty reason, `404` unknown `content_id`.
- **Storage:** append-only JSON/JSONL file on disk for the audit log and content
  status (inspectable, survives restarts, backs `GET /log`).

- **Provide:** §1 (LLM sub-section + fusion math), §2 Uncertainty Representation
  (bands, worked examples, calibration), the diagram.

  (`llama-3.3-70b-versatile`, temp 0, JSON-parsed with error fallback), and the
  `fuse(llm_score, stylometry_score)` function computing C, A, D, confidence,
  verdict — wired into `/submit`.
- **Verify:** run the labeled mini-corpus (≥5 clearly-AI, ≥5 clearly-human);
  confirm scores **vary meaningfully** (clearly-AI → conf ≥0.70 AI, clearly-human
  → conf ≥0.70 human, ambiguous → uncertain). Confirm the disagreement case
  (llm high, sty low) yields low confidence. Tune normalization constants if
  separation is weak.

### M5 — Production layer (labels, appeals, rate limit, audit log)
- **Provide:** §3 Label Design (all three variants + clause rules), §4 Appeals
  Workflow, §5 edge cases, the appeal diagram, the API contract.
- **Ask for:** `build_label(verdict, confidence, C, A)` producing the exact three
  variant strings; the `/appeal` endpoint (status → under_review, linked audit
  entry); `flask-limiter` rate limits on `/submit` and `/appeal`; the JSON/JSONL
  audit-log writer and `GET /log`.
- **Verify:** craft inputs that reach **all three** label variants and confirm
  the exact text + pct render correctly; submit then appeal an item and confirm
  status flips to `under_review` and both a decision and an appeal entry appear
  in `GET /log`; confirm rapid repeated `/submit` calls return `429`.

## Locked decisions
- **Fusion:** agreement-weighted — `C=0.6·llm+0.4·sty`,
  `confidence = D·(0.6+0.4·A)` with `D = min(1, 2.5·|C−0.5|)`.
- **Storage:** append-only JSON/JSONL file on disk.
- **Groq model:** `llama-3.3-70b-versatile` (configurable via `GROQ_MODEL`).
- **Band thresholds:** classified at `confidence ≥ 0.45`; direction at `C = 0.5`
  (calibrated empirically in M4 — see §2).
